Microsoft 365 Hardening: The 10 Settings You're Probably Missing

RP
Ryan Park Cloud & Network Engineer
December 18, 2025 7 min read

Introduction: Why Default M365 Configurations Are Dangerous

Microsoft 365 is the backbone of productivity for millions of businesses worldwide. But here is the uncomfortable truth: the default configuration that ships with every M365 tenant is optimized for ease of use, not security. And that gap between convenient defaults and hardened configuration is exactly where attackers operate.

In our work with mid-market organizations, we routinely find M365 tenants running with settings that would make any security professional wince. Legacy authentication protocols still enabled. No conditional access policies. Auto-forwarding rules silently siphoning data to external addresses. These are not edge cases — they are the norm.

The good news is that most of these vulnerabilities can be closed in a single afternoon. The bad news is that most IT teams never get around to it, either because they do not know the settings exist or because they are worried about disrupting end users. This guide walks through the 10 most commonly missed M365 hardening settings, why each one matters, and exactly how to fix it.

According to Microsoft's own research, more than 99.9% of compromised accounts did not have multi-factor authentication enabled. The defaults are not protecting you.

1 Enable Security Defaults or Conditional Access

Why it matters

Security Defaults is Microsoft's baseline package of identity security settings, including requiring MFA registration, blocking legacy authentication, and protecting privileged actions. It is designed as a safety net for organizations that have not yet implemented Conditional Access. Yet a surprising number of tenants have Security Defaults disabled — often because an admin turned them off during initial setup to avoid MFA prompts and never turned them back on.

How to fix it

Navigate to Azure Active Directory > Properties > Manage Security Defaults and toggle the setting to Yes. If your organization has Azure AD Premium P1 or P2 licenses, you should move beyond Security Defaults and implement Conditional Access policies instead, which give you granular control over sign-in requirements based on user, device, location, and risk level. At a minimum, create policies that require MFA for all users, block legacy authentication, and require compliant devices for access to sensitive applications.

2 Enforce MFA for All Admin Accounts

Why it matters

Global Administrator accounts are the master keys to your entire M365 environment. A compromised admin account gives an attacker the ability to create new accounts, export data, modify security settings, and cover their tracks. Despite this, we regularly encounter tenants where admin accounts rely on passwords alone — sometimes simple passwords at that. Even if you have Security Defaults enabled, you should explicitly verify that every account with an admin role has MFA enforced and is using a strong authentication method like a FIDO2 key or the Microsoft Authenticator app with number matching.

How to fix it

Go to Azure AD > Users > Per-user MFA (or use Conditional Access if available) and confirm that every account assigned a directory role has MFA status set to Enforced. Review the list of admin role assignments in Azure AD > Roles and administrators and remove any unnecessary role assignments. Implement the principle of least privilege: no one should have Global Administrator as their daily-driver account.

3 Disable Legacy Authentication Protocols

Why it matters

Legacy authentication protocols like POP3, IMAP, SMTP AUTH, and Basic Authentication do not support modern security features such as multi-factor authentication. Attackers know this and specifically target these protocols in password spray and credential stuffing attacks. Even if you have MFA enforced for interactive sign-ins, an attacker who obtains a user's password can authenticate through a legacy protocol and bypass MFA entirely.

How to fix it

Create a Conditional Access policy that blocks legacy authentication for all users (Azure AD > Security > Conditional Access > New policy). Set the condition to Client apps > Other clients and the access control to Block. Before enforcing, use the Report-only mode and review the sign-in logs to identify any applications or service accounts still relying on legacy protocols, then migrate them to modern authentication. Microsoft has been deprecating Basic Authentication, but tenant-level enforcement ensures nothing slips through.

4 Configure Anti-Phishing Policies in Defender for Office 365

Why it matters

The default anti-phishing policy in Microsoft Defender for Office 365 provides only baseline protection. It does not enable impersonation detection for your executives or key partners, and its thresholds for suspicious messages are set conservatively. Phishing remains the number one initial access vector for business email compromise, and sophisticated attacks that impersonate your CEO or CFO will sail past default configurations.

How to fix it

In the Microsoft 365 Defender portal > Email & collaboration > Policies & rules > Threat policies > Anti-phishing, create a custom policy. Enable mailbox intelligence and impersonation protection for your C-suite, finance team, and any employees who handle sensitive transactions. Add your top domains and partner domains to the protected list. Set the phishing email threshold to at least 2 (Aggressive) and configure actions to quarantine detected messages rather than simply adding a safety tip.

5 Enable Mailbox Audit Logging

Why it matters

Mailbox audit logging records actions taken on mailboxes, including when someone accesses another user's mailbox, sends messages as or on behalf of another user, or deletes items. While Microsoft enabled auditing by default for new tenants starting in 2019, older tenants may still have it disabled, and the default audit actions captured may not be comprehensive enough. Without audit logging, you have no forensic trail if a mailbox is compromised — you cannot tell what the attacker read, forwarded, or deleted.

How to fix it

Verify that unified audit logging is enabled in the Microsoft Purview compliance portal > Audit. Then use PowerShell to confirm mailbox-level auditing: run Get-OrganizationConfig | Format-List AuditDisabled and ensure the value is False. For critical mailboxes (executives, finance, HR), extend the audit log retention period beyond the default 90 days by assigning an E5 license or Audit (Premium) add-on, which provides one-year retention. Review the audit actions being captured and add any missing actions relevant to your compliance needs.

6 Block Auto-Forwarding to External Domains

Why it matters

One of the first things an attacker does after compromising a mailbox is set up an auto-forwarding rule to exfiltrate email to an external address. They do this quietly, and the user rarely notices because the forwarded messages still appear in their inbox. This technique is a hallmark of business email compromise attacks, and it can run for weeks or months before detection. By default, M365 allows users to configure auto-forwarding rules to any address, including external domains.

How to fix it

In the Exchange admin center > Mail flow > Remote domains, select the default remote domain (*) and set Automatic forwarding to Disabled. Additionally, create a mail flow rule (transport rule) that blocks messages where the message type is "Auto-forward" and the recipient is outside your organization. This provides a belt-and-suspenders approach. Periodically audit existing forwarding rules using PowerShell: Get-Mailbox -ResultSize Unlimited | Where-Object {$_.ForwardingSmtpAddress -ne $null} to catch any rules that were set before the policy was enacted.

7 Configure DLP Policies for Sensitive Data

Why it matters

Data Loss Prevention policies are your safety net against accidental (or intentional) leakage of sensitive information through email, Teams, SharePoint, and OneDrive. Without DLP policies, there is nothing stopping an employee from emailing a spreadsheet containing Social Security numbers, credit card data, or health records to a personal email address. Regulatory frameworks like HIPAA, PCI DSS, and state privacy laws increasingly expect organizations to have technical controls preventing data exfiltration.

How to fix it

In the Microsoft Purview compliance portal > Data loss prevention > Policies, create policies using Microsoft's built-in templates for common data types: U.S. Financial Data, U.S. PII, Health records, or custom sensitive information types relevant to your industry. Start in test mode with notifications to understand what the policy would catch before enforcing it. Configure policy tips that warn users before they share sensitive content, and set up incident reports so your security team receives alerts when DLP rules are triggered.

8 Enable Safe Attachments and Safe Links

Why it matters

Safe Attachments detonates email attachments in a sandbox environment before delivering them to the user, catching zero-day malware that signature-based scanning misses. Safe Links rewrites URLs in emails and Office documents to route clicks through Microsoft's real-time scanning service, protecting users even if a legitimate website is compromised after the email was delivered. Together, these features form a critical layer of defense against phishing and malware. They are included with Defender for Office 365 Plan 1 and above, but they must be explicitly configured.

How to fix it

In the Microsoft 365 Defender portal > Policies & rules > Threat policies, configure Safe Attachments with the action set to Dynamic Delivery (which delivers the email body immediately while the attachment is scanned) or Block for higher-security environments. For Safe Links, enable URL scanning for emails and Microsoft Teams, turn on real-time URL scanning, and enable "Do not allow users to click through to the original URL." Apply these policies to all users, not just executives.

9 Review and Restrict App Registrations and Consent

Why it matters

OAuth consent phishing has emerged as one of the most dangerous attack techniques targeting M365 environments. An attacker tricks a user into granting a malicious third-party application permissions to read their email, access their files, or send messages on their behalf. Because the user explicitly grants consent, no password is stolen and MFA is not triggered. By default, M365 allows users to consent to third-party apps requesting permissions, which means a single click from a deceived user can give an attacker persistent access to your data.

How to fix it

In Azure AD > Enterprise applications > Consent and permissions, change User consent for applications to Do not allow user consent or restrict it to verified publishers only. Implement an admin consent workflow so that users can request app access and administrators can review and approve requests. Regularly audit existing app registrations and granted permissions in Azure AD > Enterprise applications and revoke any suspicious or unnecessary grants. Pay special attention to apps with mail.read, mail.send, or files.readwrite.all permissions.

10 Implement Privileged Identity Management (PIM)

Why it matters

Standing privileged access — where admin accounts have permanent, always-on elevated permissions — is a significant security liability. If any one of those accounts is compromised, the attacker immediately has full administrative control. Privileged Identity Management eliminates standing access by making admin role assignments time-limited and approval-based. Admins must explicitly activate their role when they need it, and the activation expires after a configured period. This dramatically reduces the window of opportunity for an attacker.

How to fix it

PIM requires Azure AD Premium P2 licensing. Navigate to Azure AD > Identity Governance > Privileged Identity Management and convert all permanent admin role assignments to eligible assignments. Configure activation to require MFA and, for sensitive roles like Global Administrator, require approval from a second administrator. Set the maximum activation duration to the shortest practical period (typically 1–8 hours). Enable notifications so that all role activations are logged and key stakeholders are alerted when high-privilege roles are activated.

Bonus: Ongoing M365 Security Hygiene

Hardening your M365 tenant is not a one-time project. Microsoft continuously updates its platform, new features are added, and your organization's configuration drifts over time as employees come and go, new apps are connected, and policies are modified. To maintain a strong security posture, build these practices into your operational rhythm:

  • Run Microsoft Secure Score monthly. This built-in tool in the Defender portal benchmarks your tenant against Microsoft's recommended settings and gives you a prioritized list of improvements. Aim for incremental gains each quarter.
  • Review sign-in and audit logs weekly. Look for anomalous sign-in patterns, impossible travel events, and unfamiliar IP addresses. Azure AD Identity Protection can automate much of this if you have P2 licensing.
  • Conduct quarterly access reviews. Verify that admin role assignments, guest user access, group memberships, and app permissions are still appropriate. Remove anything that is no longer needed.
  • Test your incident response plan. Run tabletop exercises that simulate a compromised M365 account. Can your team detect it, contain it, and recover from it within your target timeframes?
  • Stay current on Microsoft security advisories. Subscribe to the Microsoft 365 Message Center and the Microsoft Security Response Center blog to stay informed about new threats, deprecated features, and configuration changes.

Security is not a destination. It is a process of continuous assessment, adjustment, and improvement. The organizations that treat M365 security as an ongoing discipline — not a project with an end date — are the ones that stay ahead of attackers.
